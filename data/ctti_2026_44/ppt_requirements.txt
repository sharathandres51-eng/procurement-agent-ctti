PLEC DE PRESCRIPCIONS TÈCNIQUES PARTICULARS — RELEVANT SECTIONS
Contract: CTTI-2026-44
Migració de la infraestructura d'equipaments als entorns de núvol híbrid

=============================================================
SECTION 3.1 — HYBRID CLOUD ARCHITECTURE REQUIREMENTS
=============================================================

3.1.1 Scope and Topology

The scope of this contract covers the migration of CTTI's on-premise data centre
workloads to a hybrid cloud platform. The target architecture must integrate:
- Hyperscaler public cloud services (Microsoft Azure and/or Amazon Web Services),
  subject to the data sovereignty constraints set out in Section 3.4.
- A private cloud environment hosted at the Barcelona Supercomputing Center (BSC)
  for workloads subject to the highest ENS classification levels or data residency
  requirements.
- Connectivity to CTTI's existing xarxa troncal de la Generalitat de Catalunya
  via dedicated private peering, not over public internet.

3.1.2 Workload Classification and Placement

The contractor must propose and document a workload classification methodology
covering at minimum:
- Sensitivity classification aligned with ENS categories: BASIC, MEDI, ALT.
- Placement rules mapping each classification to the appropriate cloud tier.
- Mandatory exclusions: workloads classified ALT under ENS that process personal
  data of Catalan citizens must not be placed on hyperscaler infrastructure outside
  the European Economic Area without explicit CTTI approval.

3.1.3 Resilience and Availability

Minimum availability requirements per service tier:

  Service tier    Min availability    RTO (critical)    RPO
  CRITICAL        99.99%              <= 4 hours        <= 1 hour
  STANDARD        99.95%              <= 8 hours        <= 4 hours
  BATCH           99.90%              <= 24 hours       <= 8 hours

All CRITICAL tier services must implement active-active or active-passive
geographic redundancy across at minimum two availability zones. Failover must
be automatic, with no manual intervention required for CRITICAL tier services.

3.1.4 Interoperability and Integration

The proposed architecture must maintain compatibility with:
- CTTI's existing identity infrastructure: Microsoft Active Directory and
  associated LDAP directory services, with federation via SAML 2.0 or OpenID
  Connect for cloud workloads.
- CTTI monitoring and observability platform: export of metrics, logs and traces
  to CTTI-operated collection points via standard protocols (OpenTelemetry,
  Prometheus remote write, syslog over TLS).
- CTTI's existing ITSM platform for incident, change and problem management
  workflows. Integration via REST API is mandatory.
- CTTI API gateway for all public-facing services; existing API contracts must
  not be broken by migration without an agreed transition period.

SECTION 3.2 — SECURITY AND REGULATORY COMPLIANCE REQUIREMENTS
=============================================================

3.2.1 Esquema Nacional de Seguretat (ENS)

The contractor must:
- Evidence current ENS certification at MEDI level or above for cloud services
  offered under this contract, or provide a credible certification roadmap with
  a binding commitment to achieve certification within 6 months of contract start.
- Deliver a complete ENS controls mapping covering all 75 applicable controls
  at MEDI level for the proposed architecture within 30 days of contract award.
- Submit quarterly ENS compliance status reports to CTTI throughout the contract.

3.2.2 GDPR and Data Protection

All processing of personal data within the scope of this contract is subject
to Regulation (EU) 2016/679 (GDPR) and Llei Orgànica 3/2018 (LOPDGDD).
The contractor must:
- Act as data processor under a data processing agreement (DPA) to be signed
  before any personal data is transferred to cloud environments.
- Implement technical measures ensuring that personal data cannot be accessed
  by hyperscaler provider staff without explicit CTTI authorisation.
- Provide CTTI with a Record of Processing Activities (ROPA) within 60 days
  of contract start, covering all processing operations within scope.
- Notify CTTI within 24 hours of detecting any personal data breach.

3.2.3 Data Sovereignty

- Storage and processing of data classified SENSIBLE must remain within the
  European Economic Area at all times. Configuration preventing cross-region
  replication to non-EEA regions must be documented and auditable.
- For workloads at the BSC private cloud, all data must remain physically
  within Catalonia. Architectural evidence of this constraint must be provided.
- The contractor must provide a data flow map covering all data in scope,
  identifying the physical region of storage for each data category.

SECTION 4.1 — TEAM ROLE REQUIREMENTS
=============================================================

4.1.1 Team Role Functions

- Responsable de compte (Account Manager): Single contractual point of contact
  for CTTI. Responsible for contract governance, escalation and strategic
  relationship management. Cannot be subcontracted.

- Cap de projecte (Project Manager): Responsible for planning, directing and
  coordinating all migration activities. Manages risk, dependencies and change
  control throughout all project phases.

- Arquitecte de solució (Solution Architect): Responsible for the end-to-end
  hybrid cloud architecture design, workload placement decisions, and technical
  coherence of the proposed solution. Must hold current cloud architect-level
  certification from at least one hyperscaler in scope.

- Enginyers de migració (Migration Engineers): Responsible for executing workload
  migration activities, including lift-and-shift, refactoring and re-platforming
  as appropriate per workload classification. Minimum two engineers assigned.

- Enginyer/a de seguretat (Security Engineer): Responsible for ENS controls
  implementation, GDPR technical measures, identity and access management
  configuration, and security monitoring integration.

- Especialista en integració (Integration Specialist): Responsible for API
  gateway configuration, identity federation, monitoring integration and ITSM
  connector implementation.

4.1.2 Minimum Profile Requirements

Solution Architect:
- Minimum 5 years of experience in enterprise cloud architecture.
- Mandatory: current certification at professional/architect level from Microsoft
  Azure (Azure Solutions Architect Expert) or AWS (AWS Certified Solutions
  Architect Professional), or equivalent from both hyperscalers.
- Prior experience with hybrid cloud deployments in Spanish public sector
  environments is strongly valued.

Security Engineer:
- Minimum 3 years of applied cloud security experience.
- Required: ENS auditor or implementer qualification, or equivalent CCN-STIC
  accreditation. CISSP or CISM certification is valued.

Project Manager:
- Minimum 5 years managing complex ICT migration projects.
- PMP or PRINCE2 Practitioner certification required.

Migration Engineers:
- Minimum 3 years of hands-on cloud migration experience.
- Azure Migrate or AWS Migration Competency certification valued.

SECTION 5 — SERVICE DELIVERY PHASES
=============================================================

5.1 Phase 1 — Pla de migració de sistemes (Migration Plan)
Duration: 3 months from contract start date.

Must include:
- Inventory and assessment of all in-scope on-premise workloads
- Workload classification per ENS category and data sensitivity
- Placement decision per workload with documented justification
- Detailed migration sequencing, dependencies and rollback plans
- Risk register with probability, impact and mitigation per risk
- Resource plan per phase with named personnel and allocation percentages
- Acceptance criteria for each migration wave

5.2 Phase 2 — Migració pilot i validació (Pilot Migration and Validation)
Duration: 4 months from Migration Plan approval by CTTI.

Migration of BATCH and lower-risk STANDARD tier workloads as proof-of-concept.
Minimum activities:
- Preparation of target cloud environments and landing zones
- Migration of agreed pilot workload set
- Performance and functional validation against pre-migration baseline
- Security and ENS controls validation for migrated workloads
- Lessons learned review and plan revision before Phase 3

5.3 Phase 3 — Migració de sistemes crítics (Critical Systems Migration)
Duration: 8 months from Phase 2 certification by CTTI.

Migration of STANDARD and CRITICAL tier workloads.
Minimum activities:
- Execution of approved migration waves for STANDARD workloads
- Migration of CRITICAL workloads with individual cutover plans per system
- Active-active/active-passive failover validation per CRITICAL service
- Decommissioning of source on-premise infrastructure following sign-off
- BSC private cloud workload deployment and integration

5.4 Phase 4 — Anàlisi post-migració (Post-migration Analysis)
Duration: 3 months following completion of Phase 3.

Minimum activities:
- Performance comparison: post-migration vs. pre-migration baseline
- Cost optimisation review and right-sizing recommendations
- ENS compliance evidence package for all migrated workloads
- Lessons learned report and architecture recommendations document

5.5 Pla de transició del servei (Service Handover)
Execution period: 2 months before contract end.

Minimum requirements:
- Minimum 4 structured knowledge transfer sessions covering architecture,
  operations, security controls and vendor management
- All documentation delivered in Catalan
- CTTI staff must perform supervised operational tasks during transition period
- 12-month post-handover warranty support period with defined SLAs
- Complete runbook and operational documentation available before handover begins

=============================================================

Key ANS indicators applicable to automatic scoring criteria (Sobre C):
- NV-DIS01: Availability of CRITICAL tier services

- NV-PUE01: Power Usage Effectiveness of proposed cloud infrastructure

Target performance: Grade 4 (Excellent) per indicator.
